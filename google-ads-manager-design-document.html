1. Executive Summary & Tool Overview

Google Ads Manager is a private, internal web application developed by Marquesi Digital exclusively for the internal use of the Vantari team. The tool provides a centralized, AI-augmented interface for managing, monitoring, and optimizing Google Ads campaigns at scale — without requiring direct access to the Google Ads web interface.

The application connects to the Google Ads API on behalf of authenticated internal users, enabling them to view campaign performance metrics, create and edit campaigns, apply bulk changes, analyze trends over custom date ranges, and leverage AI-driven recommendations for budget optimization and ad copy generation.

Important: This is a strictly internal tool. It is not distributed to the public, not published on any app store, and not offered as a service to third parties. All Google Ads accounts accessed through this tool belong to or are managed by Marquesi Digital / Vantari.

Primary Objectives

  • Reduce context-switching by consolidating campaign management in a single internal dashboard
  • Accelerate campaign optimization with AI-generated insights and recommendations
  • Automate repetitive tasks (pausing underperforming campaigns, budget adjustments) through rule-based automation
  • Enable performance comparison across date ranges without exporting data manually
  • Generate on-brand ad copies faster using Claude AI with account context

2. Target Users & Intended Use

User Base

The application is intended exclusively for internal employees of Marquesi Digital, specifically members of the Vantari team — the paid media and performance marketing department responsible for managing Google Ads accounts on behalf of clients.

Account Ownership

All Google Ads customer accounts accessed through this tool are either:

  • Owned by Marquesi Digital / Vantari directly, or
  • Client accounts for which Marquesi Digital has been granted explicit manager (MCC) access by the account owner

No access is provided to accounts without prior authorization. The tool does not allow users to self-provision access to arbitrary Google Ads accounts.

Deployment Model

The frontend is deployed as a private web application hosted on Vercel/Netlify with no public listing. The backend operates as Supabase Edge Functions accessible only via authenticated API calls. There is no public sign-up flow — access is restricted to team members who authenticate with authorized Google Workspace accounts.

Key Architectural Decisions

  • No direct API calls from the browser: All Google Ads API calls are proxied through Supabase Edge Functions. OAuth tokens and the Developer Token are never exposed to the frontend.
  • Token storage on server: Access and refresh tokens are stored encrypted in Supabase PostgreSQL, protected by Row-Level Security (RLS) policies. The browser only holds a user ID in localStorage.
  • Automatic token refresh: The proxy function checks token expiry before every API call and refreshes silently when needed, without user intervention.
  • Stateless Edge Functions: Each function invocation is independent and stateless, scaling automatically with demand.

5. Google Ads API Usage

5.1 Scopes & Permissions

All API calls are made with the https://www.googleapis.com/auth/adwords scope, which grants the ability to view and manage Google Ads accounts. The tool uses both read and write operations, as documented below.

5.2 API Endpoints Used

All requests target the Google Ads REST API v17 at https://googleads.googleapis.com/v17/.

EndpointMethodOperationPurpose
/customers/{id}/googleAds:search POST READ Execute GAQL queries to fetch campaigns, ad groups, ads, and metrics
/customers/{id}/campaigns:mutate POST WRITE Create, update, pause, or enable campaigns
/customers/{id}/adGroups:mutate POST WRITE Create, update, pause, or enable ad groups
/customers/{id}/adGroupAds:mutate POST WRITE Create, update, pause, or enable responsive search ads
/customers/{id}/campaignBudgets:mutate POST WRITE Create and update campaign budgets

GAQL Queries Executed

The application uses Google Ads Query Language (GAQL) to retrieve structured performance data. Examples of queries executed:

-- Campaign performance with budget and metrics
SELECT campaign.id, campaign.name, campaign.status,
       campaign.advertising_channel_type,
       campaign.bidding_strategy_type,
       campaign_budget.amount_micros, campaign_budget.id,
       metrics.impressions, metrics.clicks, metrics.average_cpc,
       metrics.cost_micros, metrics.conversions,
       metrics.all_conversions_value,
       metrics.search_impression_share
FROM campaign
WHERE segments.date DURING LAST_30_DAYS
ORDER BY metrics.cost_micros DESC
-- Time series for performance charts
SELECT segments.date, metrics.impressions, metrics.clicks,
       metrics.cost_micros, metrics.conversions,
       metrics.all_conversions_value
FROM campaign
WHERE segments.date DURING LAST_30_DAYS
ORDER BY segments.date

6. Application Features

6.1 Dashboard

The main dashboard displays an aggregated overview of all active campaigns for the selected Google Ads account and date range.

  • Summary KPI cards: total spend, impressions, clicks, conversions, average ROAS
  • Campaign table with sortable columns: status badge, budget, impressions, clicks, CTR, cost, conversions, ROAS
  • Quick actions per campaign: pause, enable, edit (via dropdown menu)
  • Date range selector: Last 7, 14, 30, or 90 days
  • Automatic data refresh every 5 minutes via TanStack Query stale-time configuration

6.2 Campaign Management

Full CRUD operations for the three core Google Ads entities:

📢

Campaigns

Create with name, channel type, bidding strategy (Target CPA, ROAS, Maximize Conversions), daily budget. Edit name, status, budget. Pause / enable in one click.

📁

Ad Groups

Create ad groups within campaigns, set CPC bids, manage status. View metrics per ad group with campaign context.

✏️

Responsive Search Ads

Create RSAs with up to 15 headlines (≤30 chars) and 4 descriptions (≤90 chars). Character count validation. Pin headlines to specific positions.

💰

Budgets

Create new budgets when creating campaigns. Update daily budget amounts on existing campaigns via campaignBudgets:mutate.

6.3 Analytics & Reporting

  • Time-series charts: Interactive line charts (Recharts) showing impressions, clicks, cost, conversions, CTR, and ROAS over the selected period. Metrics can be toggled individually.
  • Period comparison: Side-by-side comparison of two date ranges with absolute and percentage change indicators for 8 key metrics.
  • Per-campaign drill-down: Click any campaign to view ad group and ad-level performance breakdowns.
  • Export: Users can use browser-native print/save functionality to export charts and tables.

6.4 AI-Powered Features

Four AI modules powered by the Anthropic Claude API (model: claude-sonnet-4-6):

ModuleAPI Data Sent to ClaudeOutput

(chat)		 Top 3 and bottom 3 campaigns by ROAS (name, cost, conversions, CTR, ROAS) — no PII Natural language analysis, optimization recommendations, answers to user questions

Data minimization: Only aggregated, non-personal performance metrics are sent to the Claude API. No customer PII, no individual user data, and no full ad account details are transmitted.

6.5 Automated Rules Engine

Internal users can configure condition-based rules that run on a scheduled cron job (every 6 hours via Supabase pg_cron):

  • Condition metrics: CPA, CTR, ROAS, CPC, impressions, clicks, cost, conversions
  • Operators: greater than, less than, equal to (with configurable threshold)
  • Actions: Pause campaign, Enable campaign
  • Scope: All campaigns or a selected subset
  • Logging: Every rule execution is logged with timestamp, affected campaign IDs, and action taken

Rule executions call campaigns:mutate on the Google Ads API with status: PAUSED or status: ENABLED.

7. Security & Compliance

Credential Security

  • The Google Ads Developer Token is stored only as a server-side environment variable in Supabase Edge Functions — never in frontend code or version control
  • client_id and client_secret are server-side only; only client_id is exposed to the frontend (as required by OAuth spec)
  • OAuth access and refresh tokens are stored in Supabase with Row-Level Security policies; each user can only access their own tokens
  • The application repository is private; .env files are in .gitignore

Transport Security

  • All communication between the frontend and Supabase Edge Functions uses HTTPS/TLS 1.2+
  • All Google Ads API calls from Edge Functions use HTTPS
  • CORS headers are restricted to the application's own domain in production

Access Control

  • No public registration — only internal team members can authenticate
  • Authentication is strictly delegated to Google OAuth; the application stores no passwords
  • Supabase Row-Level Security ensures data isolation between users
  • API calls to Google Ads always include the authenticated user's access token, ensuring Google's own authorization model is respected

Data Retention & Deletion

  • Users can revoke access at any time via Google Account settings (myaccount.google.com/permissions), which invalidates the stored tokens
  • Revoking access removes the ability to make API calls; stored tokens are deleted from the database upon next access attempt
  • Campaign metric data is never persisted — it is fetched on-demand and exists only in memory during the browser session
